Overview

The Meridian Inventory Dashboard is a single-page web application for tracking stock, orders, supplier spend, demand forecasts, and (since this engagement) restocking recommendations across three warehouses (San Francisco, London, Tokyo). It serves Operations and Finance teams in English and Japanese.

The architecture is intentionally simple: a Vue 3 single-page app calls a FastAPI backend that holds all data in memory at startup, loaded from JSON files on disk. There is no database, no authentication, no message bus, no caching layer — by design (see Gaps & assumptions). This document describes the system as it exists today, after R1–R3 of this engagement; R4 is this document itself.

Audience. This document is the handoff reference for Meridian's IT team. It explains what exists and why, so that future changes — by Meridian or any future vendor — can be made with confidence.

System topology

Three runtime processes plus a static data set on disk. Browser talks to the frontend dev server, which proxies static assets; the frontend code talks directly to the backend over HTTP.

Production note. The current deployment runs all three layers on a single internal server with the dev configuration. There is no reverse proxy, no TLS termination, and CORS allows all origins. Hardening for production is out of scope of this engagement (would be a follow-up — see Gaps & assumptions).

View pattern

Every view follows the same shape — and after R1 in this engagement, this is now true of Reports.vue and Restocking.vue too. Future views should follow it. 

// 1. Reactive state with refs
const loading = ref(true), error = ref(null), items = ref([])

// 2. Composables for shared concerns
const { t, currentCurrency, translateProductName } = useI18n()
const { selectedLocation, selectedCategory, getCurrentFilters } = useFilters()

// 3. Single load function that respects filters
const loadData = async () => {
  try {
    loading.value = true
    items.value = await api.getX(getCurrentFilters())
  } catch (err) { error.value = '...' + err.message }
  finally { loading.value = false }
}

// 4. Re-run on mount and on filter changes
onMounted(loadData)
watch([selectedLocation, selectedCategory], loadData)

// 5. Computed values for derived state
const filteredItems = computed(() => ...)

Cross-cutting via composables

useFilters

Singleton-style refs for the four global filters. Imported by FilterBar.vue (writes) and by every data-bearing view (reads).

  • selectedLocation, selectedCategory, selectedStatus, selectedPeriod
  • getCurrentFilters() formats them for API calls
  • resetFilters() + hasActiveFilters for the Reset button

useI18n

Custom translation system. No external i18n library.

  • t(key, params) with placeholder substitution
  • currentLocale, currentCurrency (auto-derived from locale)
  • Domain-specific helpers: translateProductName, translateWarehouse, translateCustomerName
  • Persists chosen locale to localStorage

useAuth

Mock current-user. No real authentication today.

  • currentUser hardcoded to a sample profile
  • Different mock profile per locale (English vs Japanese)
  • Replace this with a real auth provider before production

api.js (not a composable, but cross-cutting)

Single axios-based module that every view calls.

  • Methods named getX(filters), createY(payload)
  • Filter serialization via URLSearchParams with the 'all'-skip convention
  • No interceptors — easy to add for auth tokens later

Backend

The backend lives in server/. It is a single-file FastAPI application (main.py) plus a thin data loader (mock_data.py) and a data/ directory of JSON files.

Directory layout

server/
├── main.py                # All routes, models, helpers, app config
├── mock_data.py           # Reads data/*.json into module-level lists at import
└── data/
    ├── inventory.json         # 32 SKUs across 3 warehouses
    ├── orders.json            # ~250 orders (Jan–Dec 2025)
    ├── demand_forecasts.json  # 9 SKUs with demand projections
    ├── backlog_items.json     # Outstanding shortages
    ├── purchase_orders.json   # Created via POST; starts empty
    ├── spending.json          # Aggregated cost data (procurement, operational, labor, overhead)
    └── transactions.json      # Recent transactions feed

Module-level state

At startup, mock_data.py reads each JSON file into a Python list and exposes them as module-level names (inventory_items, orders, demand_forecasts, etc.). Endpoints in main.py read and sometimes mutate these lists in place.

Mutation surfaces. The lists that grow at runtime: api_tasks (created via POST /api/tasks), purchase_orders (appended via POST /api/purchase-orders). These are lost on restart by design — see Gaps & assumptions.

Shared helpers

Two filter helpers in main.py are reused by most data endpoints. They operate on copies of the input lists and never mutate the underlying data. 

def apply_filters(items, warehouse=None, category=None, status=None) -> list
def filter_by_month(items, month) -> list  # also handles quarter strings (Q1-2025...)

Both honour the convention that 'all' means "no filter". The frontend sends 'all' when a filter is unset; the backend skips it accordingly.

Data flow

Tracing a typical request — the operator changing the warehouse filter on the Reports page — illustrates how the layers cooperate.

  • 1. UI event. Operator selects "Tokyo" in FilterBar.vue. The v-model="selectedLocation" mutates the singleton ref exposed by useFilters.
  • 2. Reactive trigger. Reports.vue has a watch([selectedPeriod, selectedLocation, selectedCategory, selectedStatus], loadData). Vue detects the change and runs loadData().
  • 3. API call. loadData invokes api.getQuarterlyReports(filters) and api.getMonthlyTrends(filters). Both pass the filter set through URLSearchParams, omitting any value of 'all'.
  • 4. Network. Axios sends GET /api/reports/quarterly?warehouse=Tokyo to http://localhost:8001.
  • 5. Backend filter. The endpoint runs apply_filters(orders, warehouse="Tokyo", ...), then computes per-quarter aggregates, sorts, and returns JSON.
  • 6. Render. Axios resolves; loadData assigns the response to quarterlyData.value; Vue's reactivity re-renders the table; computed summary stats (totalRevenue, bestQuarter) recompute.
  • 7. Locale & currency. Throughout the render, every monetary value is prefixed with {{ currencySymbol }} (a computed deriving from currentCurrency), and every label flows through t('reports.xxx').

The same pattern — composable-driven state, watch-triggered loads, centralized API client, locale-aware render — is the design contract every view follows.

Cross-cutting concerns

Internationalization

Two locales today (en, ja). All user-visible strings flow through t(); currency and number formatting flow through currencySymbol and locale-aware toLocaleString. Adding a third locale is a new file in locales/ plus an entry in the language switcher.

Global filters

Four shared filters (warehouse, category, status, period) live as singleton refs in useFilters. They drive both the API query string and any client-side narrowing. Inventory ignores status and period (no time dimension on stock); other views consume the full set.

Authentication

Currently mocked. useAuth returns a hardcoded user. There are no tokens, no protected routes, no session management. This is a deliberate gap — see Gaps & assumptions.

Error handling

Each view follows a uniform loading / error / data pattern in the template. API failures are caught in the view's loadData and rendered as an inline error banner. Backend errors return the appropriate HTTP status with a JSON detail payload (FastAPI's default).

Testing

Two parallel test suites under tests/. Both can run independently and both are CI-ready.

Backend — pytest (51 tests)

Runs against the FastAPI TestClient (no live server needed). Fast — full suite finishes in well under a second.

cd tests
uv run pytest -v
  • Endpoint smoke for inventory, orders, dashboard, demand, backlog, spending
  • Filter-combination edge cases
  • 404 paths and shape validation

E2E — Playwright (16 tests, added in R3)

Runs Chromium against the live dev servers. Covers the four critical flows declared in the proposal plus Reports and Restocking.

cd tests/e2e
npm install
npx playwright install chromium
npm test
  • Inventory · Orders · Spending · Demand · Reports · Restocking
  • Includes an explicit R1 regression test that asserts the global filter actually changes the Reports totals
  • HTML report at playwright-report/ after each run

Together: 67 tests, exercising both the API contract and the end-to-end browser experience. This is the safety net the IT team requested — every future change can be validated before going live.

Gaps & assumptions

Several things are deliberately simple in the current architecture. They are flagged here so future decisions are made with full awareness rather than assumed away.

AreaCurrent stateImplication
Persistence JSON files read at startup; mutations stored in module-level lists All runtime data (created tasks, purchase orders) is lost on backend restart. Replace mock_data.py with a database layer for production.
Authentication Mocked via useAuth; backend has no auth check Anyone with network access to the backend can read and mutate. Add auth (OIDC, session, or API key) before any external exposure.
Authorization None. Every endpoint open to all callers. Once auth is in place, role-based access (Operations vs Finance vs IT) needs an explicit policy.
CORS allow_origins=["*"] in main.py Restrict to known frontends in production.
Supplier model Restocking-originated POs use "TBD" as supplier No supplier-to-SKU mapping exists. Add one and surface a supplier picker in the Restocking flow.
Lead times Restocking uses today + 14 days as expected delivery Real lead times vary per supplier and SKU. Wire through the supplier model when added.
Stock effect of POs Creating a PO does not increment stock Correct as-is — stock changes on receipt, not on order. A receipt flow is not in scope.
Hosting Single internal server, dev configuration No reverse proxy, no TLS, no static asset CDN. Standard production hardening required.
Observability No structured logging, no metrics, no tracing Add when traffic is non-trivial. /docs is the only operational visibility today.
Why list these? The previous vendor's handoff did not flag these explicitly, which contributed to the situation Meridian found itself in. Surfacing them up front means no surprises six months from now.
