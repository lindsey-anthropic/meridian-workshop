Key Patterns

  • No Vuex/Pinia — shared state via Composition API composables
  • Centralized API client — all calls through src/api.js, base URL hardcoded to localhost:8001
  • Global filter bar — period, warehouse, category, status shared across all pages via useFilters()
  • Mock auth — hardcoded user in useAuth.js, no real session
  • i18n — custom composable, English & Japanese, locale persisted in localStorage

Composables (Shared Logic)

useFilters()

Singleton filter state shared across all pages. Period, warehouse, category, and order status. Drives API query parameters on every view.

useAuth()

Mock authentication. Returns hardcoded user profile (name, email, job title). No real token or session validation.

useI18n()

Custom i18n composable. Supports en/ja locales with localStorage persistence. Drives currency formatting (USD vs JPY).

Notable Findings

No persistent data store

All data lives in JSON files loaded into memory at startup. Any purchase orders created at runtime are lost on server restart. A real database (Postgres, SQLite) is needed for production.

Hardcoded backend URL

api.js has localhost:8001 hardcoded. Moving to any other environment requires a code change. Should use an environment variable.

CORS fully open

FastAPI is configured with allow_origins=["*"]. Acceptable for development; must be locked down before any production deployment.

Missing endpoints referenced by client

/api/tasks and /api/purchase-orders are called by the frontend but not implemented in the backend, causing 404 errors on the dashboard.

Clean separation of concerns

Frontend and backend are fully decoupled. All communication is HTTP/JSON. The composable pattern keeps shared state organized without a heavy state management library.

Filtering architecture scales well

The global filter bar feeds into a shared composable that passes parameters to every API call. Adding a new filterable view requires minimal plumbing.
