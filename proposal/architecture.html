Meridian Inventory Dashboard

Prepared for Meridian Components IT — Phase 1, Week 2 — April 2026

1. Overview

The Meridian inventory dashboard is a single-page Vue application backed by a Python (FastAPI) read-only API. It serves operations teams in three warehouses (San Francisco, London, Tokyo) for day-to-day visibility into stock levels, orders, supplier spend, demand forecasts, and a backlog queue. The system was originally delivered by a previous vendor in 2024 and has been frozen pending modernization.

It is a small system. Two services, no database, no authentication, one shared filter convention. The architectural surface area is modest. The complexity it has is mostly in the data shapes — seven JSON files representing seven loosely related domains — and in the front-end views that compose those shapes into operational reports.

Document scope

This is the institutional knowledge document the previous vendor did not leave behind. It describes the system as it is on day one of our engagement. It will be updated as we change the system through Phases 2–4.

3. System topology

Two services, one data store. Browser → Vue SPA → FastAPI → JSON files on disk.

4. Frontend

Single-page Vue 3 application. Vite serves the dev build on port 3000; production assets are static. The application bootstraps in client/src/main.js and routes through the standard Vue Router pattern into seven top-level views.

Views (client/src/views/)

  • Dashboard.vue — summary tiles and headline metrics
  • Inventory.vue — stock levels by warehouse and category
  • Orders.vue — orders with status, warehouse, month filters
  • Demand.vue — demand forecasts
  • Spending.vue — supplier spend, monthly + by-category
  • Reports.vue remediation target — quarterly + monthly trends
  • Backlog.vue — delayed-inventory queue with PO status

Composables (client/src/composables/)

  • useFilters.js — shared filter state (warehouse, category, status, month). The convention that runs through every view.
  • useI18n.js — hand-rolled translation lookup. Locale persisted in localStorage. en and ja dictionaries present.
  • useAuth.js — placeholder; not actually wired to a backend.

API client

client/src/api.js is the centralized axios wrapper with one method per backend endpoint. Most views go through api.js; Reports is the exception — it makes direct axios.get() calls to the backend, which is one of the inconsistencies R1 will address.

Pattern conventions

  • Composition API in most views; Reports is on the older Options API (cleanup target under D1).
  • v-for keys are always unique IDs — never indices.
  • Loading / error / data tri-state pattern in views that fetch.
  • Computed for derived data, refs for source data — explicit in client/CLAUDE.md.

5. Backend

Single-file FastAPI service in server/main.py (~309 lines). All endpoints are GET only — the application is read-only against the JSON data. Pydantic models define the response shapes; the same shapes are used to validate the inbound JSON files at startup via server/mock_data.py.

CORS posture

allow_origins=["*"], all methods, all headers, credentials allowed. Suitable for internal development only. Production deployment requires explicit origin allowlisting.

6. Data layer

The "database" is seven JSON files in server/data/, loaded into Python lists at module import time and held in memory for the lifetime of the process. There is no write path: changes to the JSON files require a server restart to take effect; the API itself never writes back.

Limitations of the JSON-file pattern

No concurrency control, no durability beyond what the host filesystem provides, no schema evolution story, and no query optimization. Reasonable for a small read-only demonstration; not the right shape for write paths or for scaling beyond a few thousand items per file. R2 (Restocking) is read-only and so does not require this to change. A future engagement could migrate to a relational store; we'd scope it separately.

7. The filter pattern

Every list-returning endpoint accepts the same set of optional query parameters and applies them through apply_filters() in main.py. The convention is consistent enough that it is worth documenting as a pattern — every new endpoint should follow it.

ParamSentinel for "no filter"Comparison
warehousemissing or "all"exact match on item.warehouse
categorymissing or "all"case-insensitive on item.category
statusmissing or "all"case-insensitive on item.status
monthmissing or "all"substring match on item.order_date; quarter sentinels (Q1-2025...) expanded via QUARTER_MAP

On the frontend, useFilters.js holds the same four refs and exposes getCurrentFilters() for views to forward into api.js calls. The contract is symmetric on both sides — change one, change both.

8. Request flow

Filter changes propagate through the composable, are forwarded by the api.js client, and produce a server-side in-memory filter against pre-loaded JSON.

10. Discovery findings

One finding warrants explicit mention: there are endpoints called by the frontend that do not exist on the backend.

Orphaned client endpoints

The following are defined as methods on api.js but have no route handler in server/main.py. They will fail at runtime with a 404 from FastAPI's default handler — there is no graceful client-side fallback. Disposition is part of R2 discovery.

Client methodCallsServer status
api.getTasks()GET /api/tasks404 — not implemented
api.createTask(data)POST /api/tasks404 — not implemented
api.deleteTask(id)DELETE /api/tasks/{id}404 — not implemented
api.toggleTask(id)PATCH /api/tasks/{id}404 — not implemented
api.createPurchaseOrder(data)POST /api/purchase-orders404 — not implemented
api.getPurchaseOrderByBacklogItem(id)GET /api/purchase-orders/{backlogItemId}404 — not implemented

Of note: purchase_orders.json exists, is loaded into mock_data, and its model (PurchaseOrder, CreatePurchaseOrderRequest) is defined in main.py. Everything is ready except the route registration. This looks like an unfinished feature, not dead code.

Decision required at end of Week 2: remove the orphaned client methods (and any callers), or wire up the corresponding server routes. Either way, the new Restocking work in R2 should not land alongside untriaged dead code.

12. Operational concerns

  • No authentication. Anyone reachable at :8001 can hit any endpoint. Suitable for an internal network only; production deployment requires an auth layer (proxied or in-app).
  • CORS allows everything. allow_origins=["*"] with credentials. Production needs explicit origin allowlist.
  • No rate limiting. Not a concern internally; flag for any external exposure.
  • Data refresh requires restart. Edits to JSON files are picked up only on server boot.
  • No logging or observability. No structured logs, no metrics, no traces. Adding even basic request logging is straightforward and worth doing during R3 hardening.
  • No environment-specific configuration. API_BASE_URL is hardcoded to http://localhost:8001/api in client/src/api.js. A build-time environment variable is the standard fix.

13. Onboarding a new engineer

A developer with general Vue and FastAPI experience should be able to run the system locally in under ten minutes, and should be able to answer the questions below using only this document plus the codebase.

Quick start

# Backend
cd server
uv sync
uv run python main.py    # serves on :8001

# Frontend (separate terminal)
cd client
npm install
npm run dev              # serves on :3000

# Visit http://localhost:3000 in a browser

Self-check questions

  • Where does data live? JSON files in server/data/, loaded into memory at startup by server/mock_data.py.
  • How does a filter become a query parameter? View binds to useFilters.js refs → calls api.js method with getCurrentFilters() → axios appends URL search params → FastAPI reads optional query args → apply_filters() filters the in-memory list → Pydantic-validated JSON returned.
  • Where would I add a new view? A new file in client/src/views/, register in the router, follow the loading/error/data pattern, use api.js for backend calls.
  • Where would I add a new endpoint? Pydantic model + route function in server/main.py, follow the apply_filters() convention, add a wrapper method to client/src/api.js, write a pytest case in tests/backend/.
