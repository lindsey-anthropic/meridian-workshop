2. System at a Glance

Data flow: Vue view → composable (filters) → api.js → axios GET with query params → FastAPI route → in-memory list filtering → Pydantic-validated JSON response → Vue ref / computed for render.

Reports filter inconsistency. The Reports endpoints (quarterly, monthly-trends) accept no filter parameters, even though Orders and Dashboard accept four. The Reports view nonetheless renders local filter controls outside the shared FilterBar pattern. This is one of the alignment gaps recorded for the Reports module remediation work.
Wiring gap — orphaned client methods. Six methods in client/src/api.js call backend routes that do not exist:
  • getTasks · createTask · deleteTask · toggleTask → /api/tasks (no data file, no model, no route)
  • createPurchaseOrder · getPurchaseOrderByBacklogItem → /api/purchase-orders (data and Pydantic models exist; no route)
The corresponding UI elements (TasksModal, purchase-order creation from backlog items) will fail with HTTP 404 at runtime. Symptomatic of "previous vendor work in progress" beyond the Reports module specifically. Recorded as a discovery finding for calibration with Meridian — it sits outside the current Reports remediation scope and warrants a deliberate scope decision (complete the wiring, remove the dead client code, or accept as known limitation).

5. Backend — Structure & Patterns

Single module (main.py) with co-located Pydantic models, two filtering helpers, and route handlers. Data is loaded once via mock_data.py from server/data/*.json at process start; runtime mutations (POST /tasks, POST /purchase-orders) are not persisted across restarts.

  • Filtering: centralized in apply_filters() (warehouse, category, status) and filter_by_month() (direct month or quarter via QUARTER_MAP).
  • Validation: Pydantic response_model declared on most read endpoints; spending and reports endpoints return raw dicts.
  • Errors: 404 via HTTPException on item-not-found; no auth, no rate limiting (acknowledged in server/CLAUDE.md).
  • CORS: wide open (allow_origins=["*"]) — appropriate for development, must be tightened before any deployment beyond Meridian's internal network.

6. Data Store — JSON Files

The previous vendor chose JSON-on-disk over a database. The choice is fine for the current data volume and was a reasonable accelerator during initial build. It carries explicit limits we want named so future scope decisions are made eyes-open:

  • No persistence of mutations. Anything written through the API (tasks, purchase orders) is lost on restart. R2 will introduce a "PO-issued" state transition; this constraint is what keeps it ephemeral until persistence is in scope.
  • No concurrency control. All reads operate on the same in-memory list; writes are last-write-wins.
  • Filter cost is O(N). Acceptable at <10K items; not at >100K.
  • No referential integrity. Cross-file references (e.g., orders → inventory SKUs, purchase_orders → backlog_items) are validated by convention, not by schema.
Implication for R2: the restocking algorithm reads inventory + demand-forecast + backlog cleanly from the existing endpoints. No data-store rework is required for v1. If Meridian elects cross-warehouse rebalancing (the §3.7-A4 calibration trigger), the JSON store remains adequate; the change is algorithmic, not structural.

8. Pattern Inventory — What "Aligned" Looks Like

Two repository CLAUDE.md files (client/CLAUDE.md, server/CLAUDE.md) document the conventions the previous vendor was working toward. R1 remediation aligns Reports.vue with these; future R2 and stretch work continues them.

  • Frontend: Composition API, ref/computed data layer, scoped styles, shared composables for cross-view state, centralized API client, locale strings via useI18n.
  • Backend: Pydantic response_model on read endpoints, query-param filters via shared helpers, HTTPException for not-found, lowercase comparison for case-insensitive filters.
  • Cross-cutting: ISO date strings, lowercase enum-like values for status and category, CORS-open in dev (closed in deployment).
