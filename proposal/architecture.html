1 System Overview

The Meridian Components Inventory Dashboard is a web application that gives operations staff real-time visibility into stock levels, orders, supplier spend, demand forecasts, and restocking recommendations across three warehouse locations (San Francisco, London, Tokyo).

The system consists of two processes communicating over HTTP: a Vue 3 single-page application served by Vite on port 3000, and a Python FastAPI backend on port 8001 that serves all data via a REST API. There is no database — data is loaded from JSON files at server startup and held in memory.

3 API Endpoints

All endpoints are served at http://localhost:8001/api. Interactive docs are available at http://localhost:8001/docs (Swagger UI) while the server is running.

MethodPathFilter ParamsDescription
GET /api/inventory warehouse, category All inventory items. Returns array of InventoryItem.
GET /api/inventory/{id} — Single inventory item by ID. Returns 404 if not found.
GET /api/orders warehouse, category, status, month All orders. month accepts YYYY-MM or Q1-2025 format.
GET /api/orders/{id} — Single order by ID. Returns 404 if not found.
GET /api/dashboard/summary warehouse, category, status, month Aggregate KPIs: inventory value, low-stock count, pending orders, backlog count.
GET /api/demand — All demand forecasts. No filter support (9 items covering key SKUs).
GET /api/backlog — Backlog items with has_purchase_order flag derived at request time.
GET /api/reports/quarterly warehouse, category, status, month Quarterly performance aggregated from orders (total orders, revenue, fulfillment rate).
GET /api/reports/monthly-trends warehouse, category, status, month Month-by-month order count and revenue for trend charts.
GET /api/restocking budget, warehouse, category NEW Prioritized purchase recommendations. budget=0 means no ceiling.
GET /api/spending/summary — YTD spending totals by cost category.
GET /api/spending/monthly — Month-by-month cost breakdown (procurement, operational, labor, overhead).
GET /api/spending/categories — Spending by category with percentage of total.
GET /api/spending/transactions — Recent transaction ledger (40+ entries).
Filter convention: All filter parameters are optional. Pass all or omit to return unfiltered data. String comparisons are case-insensitive.
4 Data Sources

All data lives in server/data/ as JSON files. These are loaded once at server startup into Python lists via server/mock_data.py and held in memory. Data does not persist between server restarts.

FileRecordsKey FieldsNotes
inventory.json 32 items sku, quantity_on_hand, reorder_point, unit_cost, warehouse, category Covers 3 warehouses, 5 categories. Reorder points set at ~20–40% of typical stock.
orders.json ~250 orders order_number, status, order_date, total_value, warehouse, category, items[] Full year 2025 data. Status: Delivered / Shipped / Processing / Backordered.
demand_forecasts.json 9 items item_sku, current_demand, forecasted_demand, trend Covers key SKUs only. trend: increasing / stable / decreasing. Used by Restocking algorithm.
backlog_items.json 4 items item_sku, quantity_needed, quantity_available, days_delayed, priority Unfulfilled orders. Drives "backlog" priority in Restocking.
spending.json Summary + 12 months total_procurement, total_operational, monthly[] Finance dashboard source. Static YTD aggregates.
transactions.json 40+ entries vendor, amount, category, date, warehouse Recent transaction ledger shown in Finance view.
purchase_orders.json 0 (empty) backlog_item_id, supplier_name, quantity, unit_cost Stores created purchase orders. In-memory only — resets on restart.
No database: This system uses JSON files as its data store. All data is read-only at the file level and held in memory at runtime. To update data, edit the JSON files directly and restart the server. There is no write-back path from the UI to files (exception: purchase orders are created in memory via API but not persisted).

7 How to Run

Backend

Requires Python 3.11+ and pip.

cd server
pip install -r requirements.txt
python main.py

Runs on http://localhost:8001
API docs: http://localhost:8001/docs

Frontend

Requires Node.js 18+ and npm.

cd client
npm install
npm run dev

Runs on http://localhost:3000
Backend must be running first.

E2E Tests

Requires both servers running on default ports.

cd tests/e2e
npm install
npx playwright install chromium
npm test

HTML report: npx playwright show-report

API Tests

Runs against in-memory test client — no server needed.

cd tests
pip install pytest httpx
pytest backend/ -v

51 tests, runs in ~0.2 seconds.

8 Operational Notes
TopicDetail
Data persistence None. All data is read from JSON files at startup. Any in-memory changes (e.g. created purchase orders) are lost on restart. To change data, edit the JSON files in server/data/ and restart the backend.
CORS The backend is configured to allow all origins (allow_origins=["*"]). This is intentional for local development. For any production deployment, restrict this to the frontend origin only.
Authentication There is no authentication. The mock user (John Doe / 田中太郎) is hardcoded in composables/useAuth.js. All API endpoints are publicly accessible.
Adding data Edit the relevant JSON file in server/data/. Ensure field names match the Pydantic model in server/main.py. Restart the backend to load changes.
Adding a new view Create client/src/views/NewView.vue using the Composition API pattern. Add a route in client/src/main.js. Add a nav link in client/src/App.vue. Add i18n keys in both locales/en.js and locales/ja.js.
Filter system Global filters live in composables/useFilters.js as a singleton. Any view that should respond to filter changes must import useFilters, call getCurrentFilters() in its load function, and watch the filter refs to reload on change.
Port conflicts If port 3000 or 8001 is in use, stop the conflicting process before starting the servers. On Windows: netstat -aon | findstr :3000 then taskkill /F /PID <pid>.
Browser support Tested on Chrome/Edge. The app uses standard Vue 3 features and CSS Grid/Flexbox — compatible with any modern browser (Chrome 90+, Firefox 88+, Safari 14+).
