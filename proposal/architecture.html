Meridian Components — Inventory Dashboard Architecture

Current-state overview · Prepared by Accenture · April 2026 · Based on direct codebase inspection

System Layers

Browser
Single-Page Application — Vue 3 + Vite
Runs entirely in the browser. Six views: Dashboard, Inventory, Orders, Spending, Demand, Backlog, Reports. Client-side routing via Vue Router. No page reloads.
↕
Frontend
State & API Layer — useFilters.js · api.js (Axios)
Shared filter state (warehouse, category, status, period) lives in a singleton composable. All views watch this state and re-fetch when it changes. HTTP calls go through api.js to the backend.
↕ HTTP / JSON (port 8001)
Backend
REST API — FastAPI + Uvicorn (Python)
16 endpoints across inventory, orders, dashboard, spending, demand, backlog, tasks, reports, and purchase orders. Filter parameters received as query strings; applied in-memory. Pydantic models validate all responses.
↕
Data
JSON Files — loaded at startup into memory
inventory.json · orders.json · demand_forecasts.json · backlog_items.json · spending.json · transactions.json · purchase_orders.json. No database. Changes are not persisted between restarts.

Filter System — End-to-End Flow

1 · UI
FilterBar.vue
User selects warehouse, category, status, or period
→
2 · State
useFilters.js
Singleton updates; all views watch via Vue reactivity
→
3 · HTTP
api.js (Axios)
Builds query string; GET /api/orders?warehouse=London
→
4 · Backend
main.py
apply_filters() runs in-memory list comprehension
→
5 · Response
Pydantic → JSON
Validated response updates Vue reactive ref; template re-renders

Note: Spending, Demand, Backlog, and Reports endpoints do not participate in this filter flow — they always return full datasets.

Findings — Gaps & Inconsistencies

  • R1 Reports module filters not wired. The Reports view has no connection to the shared filter state. Quarterly and monthly-trend endpoints exist on the backend but accept no filter parameters — warehouse, category, status, and period selections have no effect on Reports data.
  • R1 i18n gaps in Reports and Backlog. Several labels in these views are hardcoded English strings and do not use the useI18n composable. Tokyo warehouse staff see English-only text in these modules.
  • R1 API pattern inconsistency. Most views use Composition API + useFilters + api.js. At least one view still uses the Options API pattern from the previous vendor's earlier work — migration was incomplete at handoff.
  • Gap Spending view has no filter support. /api/spending/* endpoints return static totals regardless of warehouse or category. A London warehouse manager sees the same global numbers as a San Francisco manager.
  • Gap Purchase order UI is missing. Backend endpoints for creating and retrieving purchase orders exist and are used by the Backlog view — but there is no dedicated UI for browsing or managing purchase orders.
  • Gap No automated tests on the frontend. The tests/backend/ directory contains 51 pytest tests covering backend endpoints. There is no frontend test infrastructure.
  • Gap No persistence. All data lives in memory. Tasks created via the UI, purchase orders created via the Backlog view — all are lost on server restart. Acceptable for a demo; would need a database for production.
