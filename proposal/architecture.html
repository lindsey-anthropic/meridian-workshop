Meridian Components — Inventory Dashboard Architecture

Current-state overview · Prepared April 2026 · For handoff to Meridian IT

1. System overview

The inventory dashboard is a two-tier web application: a Vue 3 single-page frontend and a Python FastAPI backend. There is no database — all data is loaded from JSON files at server startup and held in memory. Both tiers run as local processes on the same host.

FrontendVue 3 · Composition API · Vite3000client/src/main.js
BackendPython FastAPI · Pydantic · Uvicorn8001server/main.py
DataJSON files loaded at startup (no database)—server/data/*.json

2. Request / data flow

Every view follows the same pattern. When the user changes a filter, a Vue watch() fires, api.js builds a query string from the current filter state, and the backend re-filters the in-memory data and returns a fresh response.

5. Conventions for future maintainers

  • ✓All views use Composition API (setup(), ref(), computed(), watch(), onMounted()). Do not introduce Options API (data(), methods()) in new components.
  • ✓All backend calls go through api.js. Never call axios.get() directly in a view — add a method to api.js and call that.
  • ✓Filter state lives in useFilters.js. Use getCurrentFilters() to get the current state for API calls, and watch([selectedLocation, selectedCategory, ...], loadData) to react to changes.
  • ✓All user-visible strings go through useI18n(). Add new keys to both en.js and ja.js. Currency formatting should use toLocaleString(locale, { style: 'currency', currency }) — never format currency manually.
  • ✓Backend filtering always uses list comprehensions on a copy — never mutate the global inventory_items, orders, etc. lists.
  • ✓New backend endpoints should follow the pattern: define a Pydantic response model → add response_model=List[YourModel] to the route decorator → write a test in tests/backend/.
  • ✓Data changes (adding items, changing values) are made in server/data/*.json. The server must be restarted to pick them up — data is loaded once at startup.
