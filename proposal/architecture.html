1. Technology Stack

Frontend

Vue 3 + Vite

  • Port 3000 (dev)
  • Composition API
  • Vue Router (web history)
  • Axios for HTTP
  • Custom SVG charts
  • CSS Grid / Flexbox layouts
  • i18n: English + Japanese
Backend

Python FastAPI

  • Port 8001
  • Uvicorn ASGI server
  • Pydantic data models
  • CORS: all origins (dev)
  • Auto docs at /docs
  • In-memory data (no DB)
Data

JSON flat files

  • server/data/*.json
  • Loaded at startup
  • In-memory during runtime
  • No persistence layer
  • Restart = reset to file state

All filter parameters (warehouse, category, status, month) are passed as URL query strings. The backend utility function apply_filters() in mock_data.py is shared across all endpoints. Filtering always operates on copies — the in-memory dataset is never mutated.

4. Filter System

Filters are managed by a singleton composable at client/src/composables/useFilters.js. The four filter refs (selectedPeriod, selectedLocation, selectedCategory, selectedStatus) are defined at module scope, so every component that calls useFilters() shares the same reactive state.

Filter state lives here (singleton)

useFilters() composable

selectedPeriod · selectedLocation · selectedCategory · selectedStatus

↑ written by FilterBar · read by all views ↓
FilterBar (writes) Dashboard Inventory Orders Reports Restocking Finance

Each view watches its relevant filter refs and calls loadData() on change. Filters are passed to the backend as query parameters via the shared getCurrentFilters() helper.

5. Known State and Inconsistencies

The following issues were identified during architecture review. Resolved items were addressed as part of this engagement; open items are documented for Meridian IT awareness.

  • Resolved Reports module written in Options API while rest of application uses Composition API. Migrated to Composition API as part of R1.
  • Resolved Reports filters were not connected to the global filter system. All four filters now wired and reactive.
  • Resolved Reports made direct axios.get('http://localhost:8001/...') calls with hardcoded host. Consolidated to shared api.js module.
  • Resolved Extensive debug console.log calls throughout Reports, including per-cell logging on every render. All removed.
  • Resolved Reports had no i18n — all strings hardcoded in English. Reports section added to both en.js and ja.js locale files.
  • Resolved Manual character-by-character number formatter with edge cases. Replaced with Intl.NumberFormat via toLocaleString().
  • Resolved Duplicate var i declarations in same function scope. Rewritten with const/let and array methods.
  • Resolved Bar chart height recalculated O(n²) on every render. Moved to cached computed property.
  • By design No persistence layer — all data loaded from JSON files at startup. Writes (purchase orders, tasks) are in-memory only and reset on server restart. Acceptable for current scope; a database would be required for production use.
  • Note CORS is configured to allow all origins (allow_origins=["*"]). Acceptable for internal dev; should be restricted to specific origins before any external exposure.
  • Note No authentication or authorisation layer. The application assumes a trusted internal network. Suitable for current internal deployment only.
