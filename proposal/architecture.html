Key Patterns

Filter flow

Global filter bar → useFilters singleton refs → views watch for changes → call getCurrentFilters() → pass to api.js → query params on backend.

const { selectedLocation, getCurrentFilters } = useFilters()
watch(selectedLocation, () => {
  loadData()
})
const loadData = async () => {
  const filters = getCurrentFilters()
  data.value = await api.getInventory(filters)
}

View structure

All views follow the same Composition API pattern: ref for state, computed for derived values, onMounted + watch for data loading.

setup() {
  const loading = ref(true)
  const data = ref([])

  const loadData = async () => {
    // try/catch/finally
  }

  onMounted(loadData)
  watch([...filters], loadData)
  return { loading, data, loadData }
}

Backend filter pattern

All filterable endpoints accept optional query params, pass through apply_filters() and filter_by_month() before returning data.

@app.get("/api/orders")
def get_orders(
  warehouse: Optional[str] = None,
  category: Optional[str] = None,
  status: Optional[str] = None,
  month: Optional[str] = None
):
  result = apply_filters(
    orders, warehouse, category, status
  )
  return filter_by_month(result, month)
Known Limitations (for IT)
⚠️
No database
All data lives in JSON files loaded at server startup. Changes (e.g. purchase orders created via API) are lost on restart. Adding a real database is the natural next step for production use.
⚠️
No authentication
CORS is open to all origins. The user profile is hardcoded. Not suitable for production as-is without an auth layer.
⚠️
No pagination
All endpoints return full datasets. Acceptable for current data volume (~32 SKUs, ~100 orders); will need pagination if data grows significantly.
✅
Test coverage
51 backend API tests (pytest) + 22 browser tests (Playwright). Run both before deploying any change.
