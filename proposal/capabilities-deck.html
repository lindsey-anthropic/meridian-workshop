Our Approach

Audit first. Fix what's broken. Then build what's new.

Codebase Audit

Weeks 2–5

Why this approach

We won't quote a bug count we can't verify

The RFP says "at least eight." Our clarifying questions confirmed there's no formal list — issues are in the codebase.

Committing to fix a number we haven't validated is how the previous vendor got into trouble. We commit to fix everything we find, period.

Required — R2

Restocking Recommendations

  • New view surfacing ranked purchase order recommendations
  • Inputs: current stock levels, demand forecast, operator budget ceiling
  • Budget ceiling is a hard constraint — priorities by urgency, not arbitrary order
  • Warehouse-scoped — London and Tokyo see their own picture
  • Mockup review with Meridian before development begins

Recommendations are actionable: each shows what to order, how much, and why — not raw numbers for analysts to interpret.

Relevant experience

Required — R3 + R4

Testing and Architecture Documentation

R3 — Automated Browser Testing

Meaningful coverage, not numbers

Playwright end-to-end tests covering all flows touched in this engagement: Reports filters and display, Restocking inputs and recommendations.

Each test has a plain-language description. IT can read the suite and understand what it covers without reading code. Built for CI integration if Meridian chooses to pursue it.

R4 — Architecture Documentation

What's actually running, not what was documented

The previous handoff is two pages. We will produce a current-state architecture overview — HTML format, no tooling required — based on our own audit.

Covers: system components, API surface, data flow, known technical debt, and maintainability recommendations. Written for IT, not developers.

Desired — D1–D3 (Conditional Phase)

If Meridian elects to go further

D1 — UI Modernization

Visual alignment to Meridian's website
