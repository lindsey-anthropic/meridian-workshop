The Situation

Where Meridian stands today

  • Reports module has 8+ known defects — filters not wired, i18n gaps, API inconsistencies. Unresolved since the previous vendor exited.
  • No automated test coverage. IT has blocked further changes to the system as a result.
  • Tokyo warehouse team works in English-only views. The i18n gap is an operational friction point, not a cosmetic one.
  • Restocking recommendations capability — Tanaka's ask — has not been built. The demand data exists; the feature does not.
  • The underlying platform is sound. This is a completion problem, not a rebuild problem.
Due Diligence

What we found in the codebase

Stack is solid

Vue 3 + Composition API on the frontend, Python FastAPI on the backend, JSON-based data layer. The previous vendor used the right tools. The filter system, data flow, and component patterns are coherent where they are complete.

Demand data already exists

/api/demand is live and returning forecast data. Restocking recommendations can be built on what's already there — no new data model required.

The gaps are specific

Reports module: filter components rendered but not connected to query parameters. Some views still use Options API (migration incomplete). i18n keys missing in several modules. Handoff documentation was minimal — we reconstructed the current state from code.

No test coverage

Zero automated tests were delivered. This is the structural problem — without it, every change carries approval risk. It is why IT has frozen the system.

Technical Approach

How we work

Extend, don't rewrite

The platform has a working foundation. We build in the same patterns — Vue 3 Composition API, FastAPI endpoints, the existing filter system. The codebase Meridian IT inherits will be coherent, not a patchwork of two vendors' habits.

Test coverage before big builds

R3 (testing) is established in week four — before R2 (the largest build) begins. Every subsequent change to the system is covered from that point forward.

Sequence reduces risk

Week 1–2: Architecture review and documentation. We know the system before we touch it.

Week 2–3: Reports remediation. Closes the most visible gap first.

Week 3–4: Automated tests. Unblocks IT.

Week 4–7: Restocking view. Built on proven, tested ground.

R2 — Restocking Recommendations

The capability your ops team has been waiting for

  • Data
    Built on existing demand data

    Uses /api/demand already in the system — no new forecast model required. Assumption confirmed; flag to us if data quality is a concern before we begin.

  • Logic
    Recommendation engine

    Gap between current stock and demand-adjusted target, subject to an operator-supplied budget ceiling. Budget ceiling is a required input — we do not silently override cost constraints.

  • UX
    Filterable table + CSV export

    Filter by warehouse, category, urgency. Live budget remaining counter. Export for procurement workflow.

Backend

New GET /api/restocking endpoint in FastAPI, following the same patterns as existing endpoints. Documented and tested.

Frontend

New Vue 3 view, Composition API, same layout patterns as existing views. Consistent look and feel — not a bolt-on.

Coverage

Playwright tests for the primary Restocking flow included in R3 delivery. IT can approve it from day one.
