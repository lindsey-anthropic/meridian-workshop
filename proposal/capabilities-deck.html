Our Approach

Understand before we build. Test before we ship.

  • We begin with an architecture review — our own orientation, not just a client artifact. It surfaces what the previous vendor's notes left out.
  • Every defect fix in R1 comes with a regression test. We don't mark something resolved until a test proves it.
  • R3 (browser tests) is woven through the engagement, not bolted on at the end. Tests are written alongside fixes and new features.
  • Required scope is locked before desired scope begins. No phase closes without Meridian sign-off.

R1 — Reports Remediation

Fix it properly, or don't call it fixed

The Reports module has at least eight documented defects: unwired filters, internationalization gaps, API pattern inconsistencies, and console noise. We will audit the full module before writing a single line of fix code.

Our standard: every defect gets a root-cause diagnosis, a fix, and a regression test. We will provide Meridian with a full defect log on completion — something the previous vendor never delivered.

R2 — Restocking Recommendations

A purchase order view your ops team can trust

Inputs

Current stock levels · Demand forecast · Operator-supplied budget ceiling

Output

Ranked purchase order recommendations per SKU and warehouse, with the reasoning visible to ops staff — not a black box.

Built on the existing Vue 3 frontend and FastAPI backend. The recommendation algorithm will be documented in plain language so your team can verify and trust the output.

R3 — Automated Testing

The coverage your IT team needs to say yes

We'll use Playwright — already configured in the project — to build end-to-end tests covering all filter interactions across all core views. Every filter combination. Every page.

The test suite will be documented for Meridian IT, including how to run locally and how to integrate with a CI pipeline. We don't assume prior Playwright experience on your side.

Once this is in place, your IT team has the coverage they need to approve ongoing changes with confidence.
