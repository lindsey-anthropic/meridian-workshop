The situation

A capable system, an incomplete engagement

Reports module abandoned mid-flight. The previous vendor left it using an older component pattern, bypassing the application's own API layer, with global filters that have no effect on the output.
IT has blocked all changes. No automated tests were delivered. The IT team cannot approve further development until coverage exists.
Operations is waiting. The Restocking capability R. Tanaka's team needs was never built. Every other requirement depends on IT first being unblocked.
The foundation is sound. The backend is well-structured, tested, and consistent. This is targeted remediation — not a rewrite.
Pre-proposal codebase review

What we found before writing this

Reports module

Abandoned, not incomplete

Hardcoded API URLs. Options API while all other views use Composition API. Filters ignored entirely. 14+ debug log statements in production code.

Test coverage

Backend yes, frontend zero

~50 backend tests covering core endpoints. Zero frontend tests. Reports API endpoints specifically have no coverage at all.

i18n

Infrastructure exists, partially wired

English and Japanese locale files present. Currency switching works. Reports view doesn't use any of it.

Everything else

Consistent and well-structured

Inventory, Orders, Spending, Dashboard all follow the same patterns. Backend is clean FastAPI with Pydantic validation.

Required — R1

Reports remediation

Audit-first. We enumerate every defect before touching code. The defect log is a shared artifact — signed off by Meridian before remediation begins.

What's broken today

Global filters have zero effect on Reports output
API called directly via hardcoded URLs — bypasses api.js
Options API component in a Composition API codebase
No locale-aware number or currency formatting

What we deliver

Filters integrated via shared useFilters composable
API calls routed through api.js — consistent with all views
Migrated to Composition API — maintainable by any team member
Locale-aware via existing i18n infrastructure
Required — R2

Restocking recommendations

A new view that recommends purchase orders based on current stock, demand forecast, and an operator-supplied budget ceiling. Built on the existing data the system already has.

Inputs

What the operator provides

Budget ceiling. The rest — current stock levels, demand forecast — comes from existing API data.

Output

Ranked PO recommendations

SKU, recommended quantity, estimated cost, justification (days of stock remaining, demand trend).

Integration

Fits the existing patterns

New FastAPI endpoint. New Vue view using Composition API and useFilters — consistent with the rest of the application.

Required — R3

Automated browser testing

End-to-end Playwright tests for the flows IT identified as critical. The suite lives in the repo — IT runs it before approving any future change.

Covered flows

Inventory view — load, filter by warehouse, filter by category Critical
Orders view — load, filter by status and month Critical
Reports view — filters apply post-remediation Included
Restocking — budget input, recommendations returned Critical

What IT gets

A passing test suite they can run locally or in CI. Documentation: what each test covers, how to run, how to add new tests. Up to two additional flows identified at kickoff are included without scope change.

Backend already has ~50 test cases. Our browser suite extends that to the UI layer — IT can approve changes with confidence across the full stack.

Required — R4

Architecture documentation

Delivered at the end of Phase 1 — week 1. Meridian IT should not need us to explain what they own. This document means they don't have to.

Contents
  • Component diagram — views, API client, endpoints, data
  • Data flow narrative — filter → API → response
  • Deployment notes — ports, startup, dependencies
  • Conventions — patterns future maintainers should follow
Format

Single HTML file

No dependencies. Opens in any browser. Can be printed, shared by email, or committed to the repository. Meridian IT keeps it permanently — not a document that lives on our drive.
