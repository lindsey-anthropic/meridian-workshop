05 / 12
R1 — Reports Remediation

We've already found
the problems.

Our code review of the handoff codebase identified four categories of defects before engagement start.

🔌 Filter wiring

Time Period, Warehouse, Category, and Order Status filters are partially connected. Some apply, others are ignored silently.

🌐 i18n gaps

Labels and status values hardcoded in English rather than routed through the i18n layer used in other views.

⚙️ API inconsistencies

Reports still uses the older Options API pattern — causing reactivity edge cases when filters change.

🔇 Console noise

Unhandled promise rejections and missing null checks. Non-fatal but indicative of incomplete error handling.

06 / 12
R2 — Restocking Recommendations

From stock levels
to purchase orders.

A new Restocking view that gives R. Tanaka's team exactly what they've been asking for: prioritized purchase recommendations within a budget ceiling.

How it works

Current stock ÷ demand forecast = days of coverage. Items below reorder threshold, sorted by urgency, trimmed to fit the operator's budget. No new infrastructure — uses existing API endpoints.

Input

Current stock levels (existing API)
Demand forecast (existing API)
Budget ceiling (operator input)

Output

Sortable table of recommended POs
Per-item cost and urgency score
Total within budget ceiling
